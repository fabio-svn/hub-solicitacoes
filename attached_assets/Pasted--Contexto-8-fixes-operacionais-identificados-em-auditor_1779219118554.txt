## Fix 1 — Status flow nos legacy generators

Os 2 wrappers especializados não passam por status='gerando' nem 
populam erro_geracao em falhas. Resultado: UI fica em loop sem 
feedback quando algo dá errado.

### Em gerador-assinatura.ts:

LOCALIZAR a função `gerarAssinaturaEmail`.

ADICIONAR antes do `try`:

```ts
await db.update(solicitacoesTable)
  .set({ status: "gerando", updated_at: new Date() })
  .where(eq(solicitacoesTable.id, solicitacaoId));
```

DENTRO do `catch (err)`, ANTES do `throw err`, ADICIONAR:

```ts
await db.update(solicitacoesTable)
  .set({
    status: "erro",
    erro_geracao: err instanceof Error ? err.message : String(err),
    updated_at: new Date(),
  })
  .where(eq(solicitacoesTable.id, solicitacaoId));
```

NO BLOCO DE SUCESSO (onde já tem `status: "concluido"`), GARANTIR 
que também limpa erro_geracao:

```ts
.set({
  entrega_links: [{ label: "Assinatura de E-mail", url }],
  status: "concluido",
  erro_geracao: null,  // ← adicionar
  updated_at: new Date(),
})
```

### Em gerador-cartao-boas-vindas.ts:

EXATAMENTE A MESMA LÓGICA. Adicionar status='gerando' antes do try, 
status='erro' + erro_geracao no catch, erro_geracao: null no sucesso.

## Fix 2 — Stack trace gating em admin.ts

Hoje todos os handlers retornam stack trace no JSON de resposta:
  details: err.stack?.split('\n').slice(0, 3).join('\n')

Isso expõe estrutura interna pro user. Gate por ambiente.

### Adicionar helper no topo do arquivo (após imports):

```ts
const IS_DEV = process.env.NODE_ENV !== "production";

function errorPayload(err: any, defaultMsg: string): Record<string, unknown> {
  const payload: Record<string, unknown> = {
    error: err.message || defaultMsg,
  };
  if (err.code) payload.code = err.code;
  if (IS_DEV && err.stack) {
    payload.details = err.stack.split('\n').slice(0, 3).join('\n');
  }
  return payload;
}
```

### Substituir TODOS os blocos catch que têm o pattern:

ANTES:
```ts
} catch (err: any) {
  req.log.error({ err, body: req.body }, "Erro ao ...");
  res.status(500).json({ 
    error: err.message || "Erro ao ...", 
    code: err.code, 
    details: err.stack?.split('\n').slice(0, 3).join('\n') 
  });
}
```

DEPOIS:
```ts
} catch (err: any) {
  req.log.error({ err, body: req.body }, "Erro ao ...");
  res.status(500).json(errorPayload(err, "Erro ao ..."));
}
```

APLICAR a TODOS os handlers (~11 ocorrências em admin.ts). 

ATENÇÃO: o handler que tem `res.status(err.status || 500)` (activate, 
linha ~323) deve manter o status dinâmico:
```ts
res.status(err.status || 500).json(errorPayload(err, "Erro ao ativar template"));
```

## Fix 3 — UI de erro na thankyou e solicitacao

Quando status='erro' acontece, frontend fica em polling infinito sem 
mostrar mensagem específica do erro.

### Em public/thankyou.html:

LOCALIZAR a função `iniciarPollingArtefato` (ou similar). Dentro do 
polling, quando o response retorna, verificar status:

```js
async function pollArtefato(id) {
  const res = await fetch(`/api/solicitacoes/${id}/artefato`);
  const data = await res.json();
  
  if (data.status === 'erro') {
    // Parar polling, mostrar erro
    clearInterval(pollingInterval);  // ou variável local equivalente
    document.getElementById('artefatoPolling').style.display = 'none';
    const readyEl = document.getElementById('artefatoReady');
    readyEl.style.display = 'block';
    readyEl.innerHTML = `
      <div style="color: #c53030; padding: 16px; background: rgba(197,48,48,0.08); border-radius: 8px;">
        <strong>Erro ao gerar material</strong>
        <p style="margin-top: 8px; font-size: 0.9rem;">
          Algo deu errado durante o processamento. 
          Entre em contato com o suporte ou tente novamente.
        </p>
      </div>`;
    return;
  }
  
  // ... resto da lógica existente
}
```

ATENÇÃO: adaptar pros nomes reais de variáveis/elementos no arquivo.

### Em public/solicitacao.html:

LOCALIZAR `renderEntregaAutomacao(item)`. No início, verificar 
`item.status`:

```js
async function renderEntregaAutomacao(item) {
  const card = document.getElementById('aprovacaoCard'); // ou equivalente
  
  if (item.status === 'erro') {
    const erroMsg = item.erro_geracao || 'Erro desconhecido';
    card.innerHTML = `
      <div class="entrega-card" style="border-color: #fecaca; background: #fef2f2;">
        <h3 style="color: #c53030;">Erro ao gerar material</h3>
        <p style="margin-top: 12px;">${escapeHtml(erroMsg)}</p>
        <p style="margin-top: 8px; font-size: 0.85rem; opacity: 0.7;">
          Solicitação ID: ${item.id}. Procure o suporte.
        </p>
      </div>`;
    return;
  }
  
  // ... resto da lógica existente
}
```

(usar função `escapeHtml` existente ou criar uma se não tiver — escape 
básico de <, >, ", ').

## Fix 7 — variant_value handling consistente em admin.ts

Hoje há 3 patterns diferentes pra variant_value (POST, PUT, duplicate). 
Padronizar.

### POST /art-templates (~linha 261):

JÁ ESTÁ OK (`variant_value: variant_value || null`). Manter.

### PUT /art-templates/:id (~linha 282):

MUDAR DE:
```ts
if ('variant_value' in req.body) updateData.variant_value = variant_value || null;
```

PARA:
```ts
if (variant_value !== undefined) updateData.variant_value = variant_value || null;
```

Mesma semântica, mais consistente com o resto do código.

### POST /art-templates/:id/duplicate (~linha 410):

MUDAR DE:
```ts
variant_value: variant_value !== undefined ? (variant_value || null) : source.variant_value,
```

PARA o mais claro:
```ts
variant_value: variant_value !== undefined 
  ? (variant_value || null) 
  : source.variant_value,
```

(Apenas quebra a linha pra ficar legível. Lógica é a mesma. Se já tiver 
quebrado na linha, deixa quieto.)
