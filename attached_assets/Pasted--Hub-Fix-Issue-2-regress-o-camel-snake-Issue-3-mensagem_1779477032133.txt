## FIX ISSUE 2 — Reverter auto-alias camel/snake + humanizar labels do resumo
═══════════════════════════════════════════════════════════════════════════

Modo: Economy.

### Contexto
A correção 3B anterior (alias snake_case automático em `normalizeFormDados`)
poluiu o objeto `dados`: campos como `publicoAlvo`/`prazoEntrega` ganharam
duplicatas `publico_alvo`/`prazo_entrega`, que aparecem repetidos e crus no
resumo da solicitação. Vamos reverter esse trecho e, de quebra, melhorar o
fallback de label pra qualquer campo não-mapeado ficar legível.

### Parte 1 — Reverter o auto-alias (backend)

Arquivo: `artifacts/api-server/src/routes/forms.ts`
Build necessário: `cd artifacts/api-server && pnpm run build` + restart.

LOCALIZAR e DELETAR este bloco inteiro (foi o adicionado no 3B):
```ts
  // Alias snake_case automático: pra QUALQUER campo camelCase que ainda não
  // tenha um equivalente snake_case, cria o alias derivado (ex.: nomeAssinatura
  // → nome_assinatura). Os casos irregulares continuam cobertos pelo KEY_MAP
  // acima (que já rodou). Apenas ADICIONA chaves — nunca sobrescreve nem remove
  // valores existentes, então é seguro pra quem lê em camelCase.
  for (const key of Object.keys(out)) {
    if (!/[a-z][A-Z]/.test(key)) continue;
    const snake = key.replace(/([a-z])([A-Z])/g, "$1_$2").toLowerCase();
    if (!(snake in out)) out[snake] = out[key];
  }

```

O `KEY_MAP` (os aliases explícitos como `isPrivate → is_private_key`,
`fotoPerfilDigital → foto_perfil`) PERMANECE — só o loop automático sai.

### Parte 2 — Fallback de label legível no resumo (frontend)

Arquivo: `artifacts/api-server/public/solicitacao.html`
(estático — sem build, só restart/redeploy)

LOCALIZAR (dentro de `renderDados`):
```js
        const label = (typeof DRAWER_FIELD_LABELS_FLAT !== 'undefined' && DRAWER_FIELD_LABELS_FLAT[key]) || key;
```

SUBSTITUIR por:
```js
        const label = (typeof DRAWER_FIELD_LABELS_FLAT !== 'undefined' && DRAWER_FIELD_LABELS_FLAT[key])
          || String(key).replace(/([a-z])([A-Z])/g, '$1 $2').replace(/[_-]+/g, ' ').replace(/^./, c => c.toUpperCase());
```
Assim um campo não-mapeado como `publicoAlvo` vira "Publico Alvo" em vez de
"PUBLICOALVO", e `prazo_entrega` viraria "Prazo Entrega". Os campos que JÁ têm
label bonito em `DRAWER_FIELD_LABELS_FLAT` continuam iguais (têm prioridade).

OBS: se `admin.html` renderizar o mesmo resumo com o padrão `|| key`, aplicar a
mesma substituição lá. (Procurar por `DRAWER_FIELD_LABELS_FLAT[key]) || key`.)

### Validação Issue 2
1. `pnpm run build` ok + restart.
2. Abrir uma solicitação que tenha `publicoAlvo`/`prazoEntrega`: cada campo
   aparece UMA vez só (sem o par snake duplicado).
3. O label do campo aparece legível (com espaços), não em CAIXA-COLADA.
4. Geração de arte continua funcionando (os campos que ela usa vêm do KEY_MAP,
   que não foi tocado).

### Não faça (Issue 2)
- Não remover o `KEY_MAP` (ele cobre os aliases que a arte depende).
- Não alterar o tratamento de `is_private_key` nem `palSvn`.

## FIX ISSUE 3 — Mensagem "Gerando arte" aparecendo em todo form
═══════════════════════════════════════════════════════════════════════════

Modo: Economy.

### Contexto
`window.formLockUI` mostra "Gerando sua arte... aguarde" de forma fixa em TODO
envio de formulário, inclusive nos que não geram arte. Como essa função é
genérica (compartilhada por todos os forms) e não conhece o tipo, o caminho mais
seguro é trocar por uma mensagem neutra e correta pra qualquer form.

Arquivo: `artifacts/api-server/public/utils.js`
(estático — sem build, só restart/redeploy)

LOCALIZAR:
```js
    sa.innerHTML = '<div class="form-status form-status-loading"><span class="spinner"></span><span>Gerando sua arte... aguarde</span></div>';
```

SUBSTITUIR por:
```js
    sa.innerHTML = '<div class="form-status form-status-loading"><span class="spinner"></span><span>Enviando sua solicitação... aguarde</span></div>';
```

### Validação Issue 3
1. Enviar um form SEM geração de arte (ex.: arte de divulgação, evento) →
   mensagem "Enviando sua solicitação... aguarde".
2. Enviar um form COM arte → mesma mensagem neutra (correta; o resultado da arte
   aparece depois normalmente).

### Não faça (Issue 3)
- Não tentar passar o tipo pra essa função agora (são 9+ call sites; a mensagem
  neutra resolve sem risco). Se quiser depois a mensagem específica "Gerando
  arte" só nos forms de automação, dá pra fazer num passo separado com uma flag.
