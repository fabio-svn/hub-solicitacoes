## Fix 1 — Status flow nos legacy generators

Os wrappers especializados (gerador-assinatura.ts e gerador-cartao-boas-vindas.ts)
não passam por status='gerando' nem populam erro_geracao em falhas.

### Em src/routes/gerador-assinatura.ts:

LOCALIZAR a função `gerarAssinaturaEmail`. Antes do `try {`, ADICIONAR:

```ts
await db.update(solicitacoesTable)
  .set({ status: "gerando", updated_at: new Date() })
  .where(eq(solicitacoesTable.id, solicitacaoId));
```

Dentro do `catch (err)`, ANTES do `throw err`, ADICIONAR:

```ts
await db.update(solicitacoesTable)
  .set({
    status: "erro",
    erro_geracao: err instanceof Error ? err.message : String(err),
    updated_at: new Date(),
  })
  .where(eq(solicitacoesTable.id, solicitacaoId));
```

No bloco de sucesso (onde já tem `status: "concluido"`), ADICIONAR
`erro_geracao: null` ao set:

```ts
.set({
  entrega_links: [{ label: "Assinatura de E-mail", url }],
  status: "concluido",
  erro_geracao: null,
  updated_at: new Date(),
})
```

### Em src/routes/gerador-cartao-boas-vindas.ts:

EXATAMENTE A MESMA LÓGICA. status='gerando' antes do try, 
status='erro' + erro_geracao no catch, erro_geracao:null no sucesso.

## Fix 5 — Try/catch no submit físico de cartão de visita

Em public/form-cartao-visita.html, função `submitFormFisico` (~linha 285-330).

A função digital (submitFormDigital, ~linha 335+) tem try/catch correto.
A função físico NÃO tem.

LOCALIZAR no físico onde está:
```js
const res = await fetch('/api/solicitacoes', { method: 'POST', body: formData });
const d = await res.json().catch(() => ({}));
if (res.ok) {
  // ... lógica de sucesso ...
} else {
  const errEl = document.getElementById('submitErrorFisico');
  if (errEl) { errEl.style.display = 'block'; errEl.textContent = d.error || 'Erro ao enviar. Tente novamente.'; }
  if(window.formUnlockUI)window.formUnlockUI(btn); else { btn.disabled = false; btn.textContent = 'Enviar'; }
}
```

ENVELOPAR num try/catch, padronizando com o digital:
```js

  const res = await fetch('/api/solicitacoes', { method: 'POST', body: formData });
  const d = await res.json().catch(() => ({}));
  if (res.ok) {
    // ... lógica de sucesso existente ...
  } else {
    throw new Error(d.error || 'submit_failed');
  }
} catch (e) {
  const errEl = document.getElementById('submitErrorFisico');
  if (errEl) { errEl.style.display = 'block'; errEl.textContent = 'Erro ao enviar. Tente novamente ou verifique sua conexão.'; }
  if (window.formUnlockUI) window.formUnlockUI(btn); else { btn.disabled = false; btn.textContent = 'Enviar'; }
}
```

## Fix 10 — Bug do flash de layout em solicitacao.html

Quando se abre a página de uma solicitação de automação, há um flash
do layout padrão (com dadosCard visível, solMeta, etc.) antes do layout
limpo de automação aparecer.

Causa: pageContent é mostrado ANTES de o DOM ser ajustado pro layout
correto.

Em public/solicitacao.html, função `renderPage` (~linha 614).

SUBSTITUIR o início:
```js
function renderPage(item) {
  document.getElementById('skeletonLoader').style.display = 'none';
  document.getElementById('pageContent').style.display = 'block';
```

POR:
```js
function renderPage(item) {
  // Decidir layout ANTES de tornar pageContent visível, pra evitar flash
  const isAutomation = isTipoAutomacao(item.tipo_solicitacao);
  if (isAutomation) {
    document.getElementById('solMeta').style.display = 'none';
    document.getElementById('solResponsavel').style.display = 'none';
    document.getElementById('dadosCard').style.display = 'none';
  }
  
  document.getElementById('skeletonLoader').style.display = 'none';
  document.getElementById('pageContent').style.display = 'block';
```

(o resto da função continua igual — apenas o início muda.)
