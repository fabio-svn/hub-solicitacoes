## Contexto

Após submeter um form, o user vai pra /thankyou.html. Se clica "voltar" 
no navegador, retorna pro form preenchido com UI lockada em "gerando 
arte" — o que permite resubmissões acidentais, consome rate limit e 
cria solicitações duplicadas.

Causa: usa `window.location.href` que empilha entrada no history. O 
fix é trocar por `window.location.replace()` que substitui a entrada — 
o form deixa de ser acessível via back-navigate.

Arquivos: 20 forms em artifacts/api-server/public/form-*.html

---

Pattern de substituição:
```js
// ANTES:
window.location.href = '/thankyou.html';

// DEPOIS:
window.location.replace('/thankyou.html');
```

Se em alguma linha tiver parâmetros tipo `'/thankyou.html?id=...'`, manter 
os parâmetros — só trocar o método:
```js
window.location.replace('/thankyou.html?...');
```

## Fix 2 — Safety net via pageshow event (cinto + suspensórios)

Em CADA arquivo dos forms acima, ADICIONAR um listener no início do 
`<script>` principal (logo após declaração da constante `const TIPO`):

```js
// Safety: se user volta via back-button mesmo com replace (ex: mobile 
// BFCache), recarregar a página em estado limpo
window.addEventListener('pageshow', (event) => {
  if (event.persisted) {
    try { sessionStorage.removeItem('form-' + (typeof TIPO !== 'undefined' ? TIPO : '')); } catch {}
    window.location.reload();
  }
});
```

ATENÇÃO: o nome da key sessionStorage pode variar por form. Verificar o 
key real usado em cada arquivo (procurar por `sessionStorage.setItem`). 
Se o pattern for diferente (ex: `formCartaoBoasVindas` em vez de 
`form-cartao-boas-vindas`), adaptar o `removeItem` no listener.

Se a const `TIPO` não existir no arquivo (alguns forms podem ter outro 
nome), adaptar pra usar o nome correto. O essencial: o listener fazer 
reload em caso de pageshow persisted.

## Critério de validação

1. Abrir qualquer form (ex: cartao-boas-vindas)
2. Preencher e submeter
3. Chegar em /thankyou.html
4. Clicar "voltar" no navegador
5. Validar:
   - NÃO volta pro form preenchido
   - VOLTA pra /solicitacoes.html (ou página anterior natural)
6. Sem entrada do form no history do navegador (Chrome devtools → 
   History → não vê mais form-X.html)
7. Em mobile (Safari iOS, Chrome Android), mesmo comportamento

## Não faça

- Não tocar em outras navegações (links normais, botão "fechar" do form, 
  etc.) — apenas o navigate de submit-success
- Não mexer em /thankyou.html
- Não tocar em outros arquivos não-form
- Se algum form tem flow especial (ex: cartao-visita com submit duplo 
  pra físico/digital), substituir AMBAS as ocorrências
- Não adicionar lógica de bloqueio de back-button (history.pushState ou 
  popstate intercept) — é UX agressivo, replace é suficiente
