# Cleanup v5 — auth.middleware.ts

---

## 1. MELHORIA — `requireAuth` não verifica impersonação ativa

```ts
// Quando um admin está impersonando outro usuário, req.session.user
// é substituído pelo usuário impersonado (forms.ts, POST /admin/impersonate).
// O requireAuth passa normalmente — correto ✅.
// Mas nenhum middleware expõe se a sessão está em modo impersonação,
// forçando o frontend a depender do sessionStorage para saber.
// Adicionar helper para verificar impersonação server-side:

export function isImpersonating(req: Request): boolean {
  return !!req.session?.adminOriginal;
}

// Isso permite que qualquer endpoint faça:
// if (isImpersonating(req)) { ... logar ou bloquear certas ações ... }
// E o /auth/me pode retornar o campo impersonating: true (já sugerido
// no prompt de cleanup-geral) usando este helper.
```

## 2. MELHORIA — `requireRole` repete a verificação de autenticação

```ts
// requireRole já verifica !user (401) antes de verificar o role (403).
// Isso é correto mas duplica a lógica do requireAuth.
// Para endpoints que precisam de ambos (auth + role), o padrão atual é:
//   router.get('/rota', requireAuth, requireRole('admin'), handler)
// Mas como requireRole já inclui o check de auth, o requireAuth é redundante.
// Não é um bug — é redundância segura. Adicionar comentário:

// NOTA: requireRole já inclui verificação de autenticação internamente.
// Ao usar requireRole em uma rota, não é necessário adicionar requireAuth
// antes — mas fazê-lo é inofensivo e aumenta a clareza da intenção.
```
