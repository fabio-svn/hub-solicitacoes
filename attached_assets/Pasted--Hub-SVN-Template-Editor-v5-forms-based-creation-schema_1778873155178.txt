## Etapa 1 — Schemas de form descobertos do código

Hoje o editor usa `PLACEHOLDERS_BY_TIPO` e `FIELD_OPTIONS` com mappings hardcoded e parciais. Substituir por uma **descoberta centralizada** dos schemas de cada form de automação.

### 1.1 Forms de automação conhecidos

O hub tem estes forms (visto no log de boot do servidor):

```
cartao-visita-fisico
cartao-visita-digital
cartao-boas-vindas
divulgacao-nps
convite-fp
certificado-eventos
cartao-comemorativo
assinatura-email  (gerada no fluxo principal, também conta como automação)
```

### 1.2 Descobrir schema de cada form

**Para cada form da lista acima**, inspecionar o HTML/JS correspondente no projeto e extrair:
- Lista de campos (name, type, label)
- Opções dos selects/radios (value, label)
- Quais campos são "variant-eligible" (selects com poucas opções fixas — não inputs de texto livre)

Compilar tudo em `artifacts/api-server/src/config/form-schemas.ts`:

```ts
export type FormFieldSchema = {
  name: string;              // ex: 'contrato_social'
  label: string;             // ex: 'Contrato Social'
  type: 'text' | 'email' | 'tel' | 'select' | 'radio' | 'textarea';
  options?: { value: string; label: string }[];  // pra select/radio
  required?: boolean;
};

export type FormSchema = {
  tipo: string;              // 'cartao-boas-vindas'
  label: string;             // 'Cartão de Boas-vindas'
  description?: string;
  fields: FormFieldSchema[];
  // Campos virtuais/computados que existem como placeholder mas não como input
  // Ex.: 'contrato_label' é derivado de 'contrato_social'
  computed?: { name: string; label: string; derived_from?: string }[];
};

export const FORM_SCHEMAS: Record<string, FormSchema> = {
  'cartao-boas-vindas': {
    tipo: 'cartao-boas-vindas',
    label: 'Cartão de Boas-vindas',
    fields: [
      { name: 'nome_cliente',    label: 'Nome do cliente',     type: 'text', required: true },
      { name: 'nome_assinatura', label: 'Nome para assinatura', type: 'text', required: true },
      { name: 'unidade',         label: 'Unidade',              type: 'text', required: true },
      {
        name: 'contrato_social', label: 'Contrato Social', type: 'select', required: true,
        options: [
          { value: 'svn-investimentos', label: 'SVN Investimentos' },
          { value: 'svn-capital',       label: 'SVN Capital' },
          { value: 'svn-connect',       label: 'SVN Connect' },
        ],
      },
      {
        name: 'is_private_key', label: 'Cliente Private?', type: 'radio', required: true,
        options: [
          { value: 'padrao',  label: 'Não' },
          { value: 'private', label: 'Sim' },
        ],
      },
    ],
    computed: [
      { name: 'contrato_label', label: 'Contrato (label)', derived_from: 'contrato_social' },
    ],
  },
  // ... preencher os outros 7 forms inspecionando o código deles
};
```

**Importante:** se algum form não estiver implementado ainda no projeto, ainda assim criar entrada placeholder com a estrutura conhecida e marcar `description: 'Form ainda não implementado'`.

### 1.3 Endpoint pra expor schemas ao frontend

```ts

  ...
]

// GET /api/admin/form-schemas/:tipo
// Retorna schema completo do form específico
```

Frontend do editor consome esses endpoints em vez de constantes hardcoded.

## Etapa 2 — Criação de template baseada em form

### 2.1 Fluxo novo na view "create"

Substituir a tela atual de criação:

**Antes:**
```
Tipo de arte: [select com tipos + "criar novo" ▾]
Nome: [_______________]
Começar de: [● em branco] [○ clonar de...]
```

**Depois:**
```
Para qual formulário esta arte será usada? *
[Selecione um formulário                            ▾]
  ├─ Cartão de Boas-vindas
  ├─ Cartão de Visita Físico
  ├─ Cartão de Visita Digital
  ├─ Divulgação NPS
  ├─ Convite FP
  ├─ Certificado de Eventos
  ├─ Cartão Comemorativo
  └─ Assinatura de E-mail

Nome do template *
[Ex: "Cartão minimalista v2"                          ]

Começar a partir de:
( ) Template em branco (canvas 1080×1080)
(●) Clonar de existente
    [Select de templates existentes pro form escolhido ▾]
```

### 2.2 Validações

- Form é obrigatório — sem ele, botão "Criar" desabilitado
- Ao selecionar "Clonar de existente", o dropdown filtra apenas templates do mesmo tipo do form escolhido
- Se o form escolhido ainda não tem nenhum template (caso comum nas primeiras vezes), opção "Clonar" fica desabilitada e só "Template em branco" disponível

### 2.3 Remover opção "Criar novo tipo"

Tipos não são mais livremente criáveis pela UI. Se vocês criarem um novo form de automação no futuro, ele entra no `FORM_SCHEMAS` via código (parte da implementação do form), e aí aparece automaticamente no dropdown.

Isso garante que **todo template está vinculado a um form real**, evitando templates órfãos.

## Etapa 3 — Variant options dinâmicas por form

### 3.1 Problema atual

Hoje, ao configurar uma layer como `variant`, as opções vêm de `FIELD_OPTIONS` hardcoded, que mistura campos de vários forms. Resultado: ao editar template de Cartão Comemorativo, aparecem opções do Cartão de Boas-vindas (que não são as certas).

### 3.2 Solução

Ao editar um template, **buscar o schema do form correspondente** (`GET /api/admin/form-schemas/:tipo`) e filtrar variants apenas pelos campos daquele form:

```js
// Após carregar template:
const schema = await fetch(`/api/admin/form-schemas/${currentTemplate.tipo}`).then(r => r.json());
formSchema = schema;

// Lista de placeholders disponíveis (= campos + computed do form):
const placeholders = [
  ...schema.fields.map(f => f.name),
  ...(schema.computed || []).map(c => c.name),
];

// Campos eligíveis pra variant (apenas selects/radios com opções):
const variantEligibleFields = schema.fields.filter(f =>
  (f.type === 'select' || f.type === 'radio') && f.options?.length
);
```

### 3.3 UI da configuração de variant

Ao selecionar tipo `variant` numa layer de imagem:

```
Tipo da imagem: [variant ▾]

Campo variável: [Selecione um campo            ▾]
  ├─ contrato_social (Contrato Social)
  ├─ is_private_key (Cliente Private?)
  └─ ... apenas campos variant-eligible do form

Variantes:
  SVN Investimentos    [URL                              ] [📁]
  SVN Capital          [URL                              ] [📁]
  SVN Connect          [URL                              ] [📁]
```

(O ícone `📁` é o botão de "Selecionar do asset library" — ver Etapa 5.)

As opções vêm direto do schema do form. Os labels são humanizados (vêm do `options[].label`).

## Etapa 5 — Upload de imagens + Asset Library

### 5.1 Schema

```sql
CREATE TABLE art_assets (
  id SERIAL PRIMARY KEY,
  filename VARCHAR(300) NOT NULL,         -- nome original do arquivo
  storage_key VARCHAR(500) NOT NULL UNIQUE,  -- key no R2: 'assets/2026-05/abc123.png'
  url VARCHAR(500) NOT NULL,              -- URL pública
  mime_type VARCHAR(100) NOT NULL,
  size_bytes BIGINT NOT NULL,
  width INTEGER,
  height INTEGER,
  uploaded_by INTEGER REFERENCES users(id),
  uploaded_at TIMESTAMP NOT NULL DEFAULT NOW(),
  -- tracking de uso (atualizado em cada save de template via scan)
  used_in_template_ids INTEGER[] DEFAULT '{}',
  last_used_at TIMESTAMP
);

CREATE INDEX idx_art_assets_uploaded_at ON art_assets(uploaded_at);
CREATE INDEX idx_art_assets_used_template ON art_assets USING GIN(used_in_template_ids);
```

### 5.2 Endpoints

| Endpoint | Descrição |
|---|---|
| `POST /api/admin/assets/upload` | Recebe `multipart/form-data` com `file`. Faz upload pro R2 (path `assets/YYYY-MM/{nanoid}-{filename}`), salva no DB, retorna `{ id, url, filename }` |
| `GET /api/admin/assets` | Lista assets com filtros: `?orphan=true` (não usado em nenhum template), `?uploaded_before=YYYY-MM-DD`, paginação |
| `DELETE /api/admin/assets/:id` | Remove do R2 + DB. Bloqueia se `used_in_template_ids` não está vazio (precisa unbind antes) |
| `POST /api/admin/assets/scan-usage` | Job manual: lê todos os templates, atualiza `used_in_template_ids` e `last_used_at` |

Todos com `requireAdmin`.

### 5.6 Sanity checks no upload

- Tamanho máximo: 10 MB
- Tipos aceitos: PNG, JPG, WEBP, SVG
- Largura/altura máximas: 8000×8000 (Sharp lê metadata após upload e rejeita se exceder)
- Nome de arquivo sanitizado (remove caracteres especiais)

### 5.7 Tracking de uso

Função `scanAssetUsage()`:

```ts
async function scanAssetUsage() {
  // 1. Pega todos os templates ativos + inativos
  const templates = await db.select().from(artTemplates);

  // 2. Reset uso
  await db.update(artAssets).set({ used_in_template_ids: [], last_used_at: null });

  // 3. Pra cada template, scan as URLs no config
  for (const t of templates) {
    const urls = extractUrlsFromConfig(t.config);
    for (const url of urls) {
      // Encontra asset por URL
      const asset = await db.query.artAssets.findFirst({ where: eq(artAssets.url, url) });
      if (asset) {
        await db.update(artAssets)
          .set({
            used_in_template_ids: sql`array_append(used_in_template_ids, ${t.id})`,
            last_used_at: t.updated_at,
          })
          .where(eq(artAssets.id, asset.id));
      }
    }
  }
}

function extractUrlsFromConfig(config: ArtTemplate): string[] {
  const urls = new Set<string>();
  // bg
  if (config.bg.type === 'static') urls.add(config.bg.url);
  else Object.values(config.bg.variants).forEach(u => urls.add(u));
  // layers de imagem
  for (const layer of config.layers) {
    if (layer.type === 'image') {
      if (layer.source.type === 'static' && layer.source.url) urls.add(layer.source.url);
      if (layer.source.type === 'variant') Object.values(layer.source.variants || {}).forEach(u => urls.add(u));
    }
  }
  return Array.from(urls);
}
```

Rodar `scanAssetUsage()`:
- Manualmente via endpoint (admin clica em "Recomputar uso")
- Automaticamente após cada save de template
- Como cron diário (opcional)

### 5.8 Considerações de armazenamento

Os assets ficam no mesmo bucket R2 que já usam, mas no prefix `assets/`. Esses são diferentes dos artefatos gerados (que ficam em `cartoes-boas-vindas/`, `assinaturas/`, etc.) — não confundir as duas coisas.

R2 não cobra por storage até 10GB no plano gratuito. Pro caso de uso normal (dezenas/centenas de imagens, alguns MB cada), está num custo desprezível.

### Migração

- Templates existentes têm URLs antigas (não vão estar na tabela `art_assets` ainda)
- Solução: na primeira execução do `scanAssetUsage`, **opcional**: tentar fazer match das URLs com assets já no R2 e criar entradas retroativas no DB pra esses (best effort). Senão, ficam como "external URLs" não-gerenciáveis pela library — vão funcionar normalmente, só não dá pra deletar pela UI.
