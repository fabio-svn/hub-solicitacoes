## #1 — Ctrl + scroll para zoom no Editor de Templates

Adicionar wheel listener no canvas que, quando `event.ctrlKey || event.metaKey` está ativo, aplica zoom no template:

```js
document.getElementById('canvasScroll').addEventListener('wheel', (e) => {
  if (!(e.ctrlKey || e.metaKey)) return;  // só age com Ctrl/Cmd pressionado
  e.preventDefault();
  // deltaY positivo = scroll pra baixo = zoom out
  // deltaY negativo = scroll pra cima = zoom in
  const delta = -e.deltaY * 0.001;  // sensibilidade
  canvasScale = Math.max(0.1, Math.min(2, canvasScale + delta));
  applyCanvasScale();
  updateZoomLabel();
}, { passive: false });
```

**Detalhe importante:** o `{ passive: false }` é necessário pra poder usar `preventDefault()` no wheel event (caso contrário a página rola junto). Sem `Ctrl/Cmd` pressionado, scroll continua funcionando normal (rolar dentro do canvas).

Persistir `canvasScale` em `sessionStorage` (já deve estar no v3).

## #4 — Fluxo de geração no submit do form

### 4.1 Comportamento desejado

Hoje (suposto):
1. Usuário submete form → backend processa
2. Frontend **redireciona imediatamente** pra Thank You page
3. Thank You page mostra "Gerando..." com spinner
4. Quando artifact_url aparece, mostra botão de download

Desejado:
1. Usuário clica em Enviar
2. **Permanece no form** com toda a UI travada (botão Enviar desabilitado, inputs disabled)
3. Aparece **inline no rodapé do form** uma área com:
   - Spinner pequeno
   - Texto "Gerando sua arte... aguarde"
4. Backend responde (sucesso ou erro)
5. **Se sucesso:** redireciona pra Thank You page (que já carrega com URL pronto)
6. **Se erro:** spinner some, aparece mensagem de erro vermelha "Houve um problema ao gerar sua arte. Tente novamente ou contate o suporte." + reabilita o form

### 4.2 Implementação

No JS do form (`solicitacao-form.js` ou similar):

```js
async function handleSubmit(e) {
  e.preventDefault();
  const form = e.target;
  const submitBtn = form.querySelector('button[type="submit"]');
  const statusArea = document.getElementById('formStatusArea'); // criar div no HTML

  // Trava UI
  submitBtn.disabled = true;
  form.querySelectorAll('input, select, textarea, button').forEach(el => el.disabled = true);

  // Mostra status
  statusArea.innerHTML = `
    <div class="form-status form-status-loading">
      <span class="spinner"></span>
      <span>Gerando sua arte... aguarde</span>
    </div>
  `;
  statusArea.style.display = 'block';

  try {
    const res = await fetch('/api/solicitacoes', {
      method: 'POST',
      headers: { 'Content-Type': 'application/json' },
      credentials: 'include',
      body: JSON.stringify(buildPayload(form)),
    });

    if (!res.ok) {
      const errData = await res.json().catch(() => ({}));
      throw new Error(errData.error || `HTTP ${res.status}`);
    }

    const data = await res.json();

    // Sucesso → redireciona
    window.location.href = `/solicitacoes/${data.id}`;

  } catch (err) {
    // Erro → mostra mensagem e reabilita
    statusArea.innerHTML = `
      <div class="form-status form-status-error">
        <span>⚠️ Houve um problema ao gerar sua arte. Tente novamente ou contate o suporte.</span>
        <small>${err.message}</small>
      </div>
    `;
    submitBtn.disabled = false;
    form.querySelectorAll('input, select, textarea, button').forEach(el => el.disabled = false);
  }
}
```

CSS:

```css
.form-status { padding: 16px; border-radius: 10px; margin-top: 16px; display: flex; align-items: center; gap: 12px; }
.form-status-loading { background: #f5f3f0; color: #221b19; }
.form-status-error { background: #fef2f2; color: #991b1b; border: 1px solid #fecaca; flex-direction: column; align-items: flex-start; }
.form-status-error small { color: rgba(153,27,27,0.65); font-size: 0.75rem; }
.spinner {
  width: 16px; height: 16px; border: 2px solid #ccc; border-top-color: #9f3f37;
  border-radius: 50%; animation: spin 0.7s linear infinite;
}
@keyframes spin { to { transform: rotate(360deg); } }
```

### 4.3 Backend: garantir que `POST /api/solicitacoes` só responde após geração completa

O endpoint deve aguardar `renderFromTemplate` + upload R2 + atualização do DB antes de retornar 200. Se algum desses passos falhar, retornar 500 com mensagem clara no body (o fallback do v4 já protege contra crash, mas pro caso do form-submit, propagar o erro pro frontend pra poder mostrar a mensagem certa).

**Considerar:** se a geração demora >10s, o frontend pode dar timeout em algumas configurações de proxy. Tipicamente Sharp leva <2s, então não deve ser problema, mas vale monitorar.

## #5 — "Ver solicitação" leva pra página em branco

Bug. O botão na Thank You page que diz "Ver solicitação" deveria levar pra `/solicitacoes/{id}` (página de detalhes da solicitação), mas leva pra uma página em branco.

### 5.1 Diagnóstico

Conferir:
1. **A URL gerada está correta?** Inspecionar o `href` do botão. Deve ser tipo `/solicitacoes/123` (não `/solicitacao/123`, sem o `s`, ou path errado).
2. **Existe rota servindo esse path?** Verificar se há um handler pra `GET /solicitacoes/:id` no Express.
3. **Existe HTML pra renderizar?** Pode ser que a rota exista mas serve um HTML que dá fetch nos dados — se o fetch falha, página fica em branco.

### 5.2 Provável causa

A causa mais comum desse tipo de bug: o redirect foi configurado pra um path que existe no roteamento do dashboard mas não fora dele. Conferir que o usuário consegue acessar `/solicitacoes/{id}` direto pela barra do navegador (cole a URL e vê se aparece a página). Se sim, problema é só no `href` do botão.

### 5.3 Fix sugerido

No botão da Thank You page:

```html
<a href="/solicitacoes/{{solicitacao_id}}" class="btn btn-primary">Ver solicitação</a>
```

Garantir que `solicitacao_id` é o valor correto (o ID retornado pela POST que criou a solicitação).

Se a URL estiver correta e ainda assim a página fica em branco, **abrir DevTools → Console na página em branco** e ver o erro. Pode ser falha de auth, fetch de dados que falha, etc.

## #7 — Permitir deletar único template de categoria sem solicitações

### Problema atual

Hoje, ao tentar deletar um template, aparece "Este é o único template da categoria. Crie um novo antes de deletar este." Mesmo quando a categoria é nova, sem solicitações criadas — o que torna impossível abandonar uma categoria de teste.

### Regra nova

Backend permite deletar baseado em **se há solicitações de produção dependendo daquela categoria**:

```ts
// DELETE /api/admin/art-templates/:id
async function deleteTemplate(req, res) {
  const id = +req.params.id;
  const template = await db.query.artTemplates.findFirst({ where: eq(artTemplates.id, id) });
  if (!template) return res.status(404).json({ error: 'Template não encontrado' });

  // Conta outros templates do mesmo tipo
  const otherTemplates = await db.select({ count: count() })
    .from(artTemplates)
    .where(and(eq(artTemplates.tipo, template.tipo), ne(artTemplates.id, id)));
  const otherCount = otherTemplates[0].count;

  if (otherCount === 0) {
    // É o único da categoria. Conferir se há solicitações dependendo dele.
    const solicitacoesDoTipo = await db.select({ count: count() })
      .from(solicitacoes)
      .where(eq(solicitacoes.tipo, template.tipo));
    const solicCount = solicitacoesDoTipo[0].count;

    if (solicCount > 0) {
      return res.status(400).json({
        error: `Este é o único template da categoria "${template.tipo}" e existem ${solicCount} solicitações dessa categoria. Crie outro template antes de deletar.`,
      });
    }
    // Sem solicitações → pode deletar livremente (categoria órfã/teste)
  }

  await db.delete(artTemplates).where(eq(artTemplates.id, id));
  res.json({ ok: true });
}
```

### Frontend

Quando admin clica em Deletar, mostrar confirmação contextualizada:

```js
async function confirmDelete(template) {
  const isOnly = await checkIfOnlyTemplate(template.id);
  let message;
  if (isOnly) {
    message = `Você está deletando o único template da categoria "${template.tipo}". Esta categoria ficará sem template e qualquer form associado a ela vai falhar até você criar outro. Confirma?`;
  } else {
    message = `Deletar template "${template.name}"?`;
  }
  if (!confirm(message)) return;
  // ... DELETE call
}
```

A mensagem de erro do backend (quando há solicitações) é exibida via toast vermelho.
