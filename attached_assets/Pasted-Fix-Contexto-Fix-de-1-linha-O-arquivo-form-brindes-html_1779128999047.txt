Fix 5:
## Contexto

Vulnerabilidade em produção: POST /api/webhook/clickup aceita qualquer 
requisição sem autenticação. Atacante que conhece um ID de solicitação 
pode mudar status arbitrariamente.

Correção: validar X-Signature contra body raw com HMAC SHA256.

A var CLICKUP_WEBHOOK_SECRET deve existir no .env (configurada ao criar 
o webhook no ClickUp). Se não existir, adicione ao .env e ao .env.example.

Spec: https://clickup.com/api/developer-portal/authentication/

## Arquivo

artifacts/api-server/src/routes/webhook.ts

## Tarefa

1. Importar crypto no topo do arquivo:
     import crypto from 'crypto';

2. Pra preservar body raw, usar express.raw na rota:

     router.post(
       '/clickup',
       express.raw({ type: 'application/json' }),
       async (req, res) => {
         const signature = req.headers['x-signature'] as string | undefined;
         const secret = process.env.CLICKUP_WEBHOOK_SECRET;

         if (!secret) {
           logger.error('CLICKUP_WEBHOOK_SECRET não configurado');
           return res.status(500).json({ error: 'webhook_not_configured' });
         }
         if (!signature) {
           return res.status(401).json({ error: 'missing_signature' });
         }

         const expected = crypto
           .createHmac('sha256', secret)
           .update(req.body)
           .digest('hex');

         if (!crypto.timingSafeEqual(
           Buffer.from(signature, 'hex'),
           Buffer.from(expected, 'hex')
         )) {
           return res.status(401).json({ error: 'invalid_signature' });
         }

         const payload = JSON.parse(req.body.toString('utf-8'));

         // ... lógica original aqui, usando `payload` em vez de req.body
       }
     );

3. Adicione ao .env.example:
     CLICKUP_WEBHOOK_SECRET=

## Critério de validação

1. Sem signature → 401:
   curl -X POST http://localhost:3000/api/webhook/clickup \
     -H "Content-Type: application/json" \
     -d '{"event":"taskStatusUpdated"}'

2. Signature inválida → 401:
   curl -X POST http://localhost:3000/api/webhook/clickup \
     -H "Content-Type: application/json" \
     -H "X-Signature: deadbeef0000000000000000000000000000000000000000000000000000000" \
     -d '{"event":"taskStatusUpdated"}'

3. Real: confira que o secret no ClickUp (Settings → Integrations → 
   Webhooks) bate com CLICKUP_WEBHOOK_SECRET. Dispare um evento real 
   e veja o log do handler executando.

## Não faça

- Não altere a lógica de update de status, só adicione a validação antes
- Não logue secret ou signature
- Não desabilite a validação em desenvolvimento sem flag explícita 
  (ex.: WEBHOOK_SKIP_AUTH=1)

Fix 7:
## Contexto

3 bugs em produção com mesma raiz: forms enviam payload `dados` em 
camelCase com valores raw, backend espera snake_case normalizado.

Bugs cobertos:

1. divulgacao-nps: form envia `modeloArte: "com-foto"`. Backend procura 
   `modelo_arte`. Variant lookup falha.

2. cartao-boas-vindas: form envia `isPrivate: "sim"`. Backend procura 
   `is_private_key: "private"`. Variant lookup falha. Hoje só funciona 
   porque o legacy generator (gerador-cartao-boas-vindas.ts) intercepta 
   antes.

3. cartao-visita-digital: suspeita similar.

A fix é normalização no backend, ANTES da persistência e antes do 
art-generator. Não mexer nos forms.

## Arquivo

artifacts/api-server/src/routes/forms.ts

(Opcional: criar `src/services/payload-normalizer.ts` se preferir 
separar do forms.ts.)

## Tarefa

1. Função utilitária:

   /**
    * Normaliza `dados` de camelCase (formato dos forms) para snake_case 
    * com valores normalizados (formato esperado por art-generator, 
    * clickup.ts e template-renderer). Retorna NOVO objeto. Não muta.
    */
   export function normalizeFormDados(
     tipo: string,
     dados: Record<string, unknown>
   ): Record<string, unknown> {
     const out = { ...dados };

     // 1. Tradução de chaves camelCase → snake_case
     const KEY_MAP: Record<string, string> = {
       isPrivate: 'is_private_key',
       modeloCartao: 'modelo_cartao',
       modeloArte: 'modelo_arte',
       contratoSocial: 'contrato_social',
       nomeCliente: 'nome_cliente',
       nomeAssinatura: 'nome_assinatura',
       nomeCompleto: 'nome_completo',
     };
     for (const [camel, snake] of Object.entries(KEY_MAP)) {
       if (camel in out && !(snake in out)) {
         out[snake] = out[camel];
       }
     }

     // 2. Tradução de valores
     if (out.is_private_key === 'sim') out.is_private_key = 'private';
     else if (out.is_private_key === 'nao') out.is_private_key = 'padrao';

     // palSvn{N}: garantir 'Sim'/'Não' capitalizado
     for (let i = 1; i <= 4; i++) {
       const k = 'palSvn' + i;
       const v = out[k];
       if (typeof v === 'string') {
         const lower = v.toLowerCase();
         if (lower === 'sim') out[k] = 'Sim';
         else if (lower === 'nao' || lower === 'não') out[k] = 'Não';
       }
     }

     return out;
   }

2. No handler POST /api/solicitacoes, ANTES de persistir no banco, 
   chamar createClickUpTask ou chamar o art-generator:

     const dadosOriginal = typeof req.body.dados === 'string'

     const dados = normalizeFormDados(tipo_solicitacao, dadosOriginal);

3. IMPORTANTE: persistir no banco o objeto NORMALIZADO. Não persistir 
   o camelCase original.

4. Cruze referências com src/services/art-template.ts e 
   src/services/art-generator.ts: confirme que esses módulos esperam as 
   chaves snake_case que estamos gerando. Se algum espera camelCase,
