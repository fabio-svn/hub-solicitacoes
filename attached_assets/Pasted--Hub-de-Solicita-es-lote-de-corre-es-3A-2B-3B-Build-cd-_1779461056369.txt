## PROMPT 2B — Limpar arquivo órfão no R2 se o insert no banco falhar
═══════════════════════════════════════════════════════════════════════════

Modo: Economy.

### Problema
Em `POST /solicitacoes`, dentro do loop de upload: se `uploadToR2` der certo mas
o `db.insert(arquivosTable)` seguinte falhar, o arquivo fica no R2 sem referência
no banco (lixo + custo). O `catch` atual só loga. O `deleteFromR2` já está
importado e aceita a própria URL retornada pelo upload.

### Arquivo: `artifacts/api-server/src/routes/forms.ts`

LOCALIZAR (dentro do `POST /solicitacoes`):
```ts
      for (const file of files) {
        try {
          const url = await uploadToR2(file, solicitacao.id, file.fieldname);
          await db.insert(arquivosTable).values({
            solicitacao_id: solicitacao.id,
            campo: file.fieldname,
            url_r2: url,
            nome_original: file.originalname,
          });
          arquivosMap[file.fieldname] = url;
        } catch (uploadErr) {
          logger.error({ err: uploadErr }, "R2 upload failed, continuing");
        }
      }
```

SUBSTITUIR por:
```ts
      for (const file of files) {
        let uploadedUrl: string | null = null;
        try {
          uploadedUrl = await uploadToR2(file, solicitacao.id, file.fieldname);
          await db.insert(arquivosTable).values({
            solicitacao_id: solicitacao.id,
            campo: file.fieldname,
            url_r2: uploadedUrl,
            nome_original: file.originalname,
          });
          arquivosMap[file.fieldname] = uploadedUrl;
        } catch (uploadErr) {
          logger.error({ err: uploadErr }, "R2 upload failed, continuing");
          // Se o upload subiu mas o registro no banco falhou, o arquivo ficaria
          // órfão no R2 — limpa pra não gerar lixo/custo. deleteFromR2 já engole
          // o próprio erro, então não precisa de try/catch aqui.
          if (uploadedUrl) {
            await deleteFromR2(uploadedUrl);
            logger.warn({ url: uploadedUrl }, "Arquivo órfão removido do R2 após falha no insert");
          }
        }
      }
```

### Validação 2B
1. `pnpm run build` ok.
2. Fluxo normal: enviar solicitação com anexo → arquivo no R2 + linha em
   `arquivos` (sem regressão).
3. Comportamento de órfão: difícil simular sem forçar erro, mas confirmar que o
   `deleteFromR2` está sendo chamado apenas no caminho de falha (não no sucesso).

### Não faça (2B)
- Não mover o upload pra fora do loop nem mudar a ordem insert/upload do fluxo.
- Não envolver `deleteFromR2` em try/catch (ele já trata o próprio erro).
- Não tocar nos `db.update` de título/ClickUp mais abaixo.

## PROMPT 3B — Alias snake_case automático em normalizeFormDados
═══════════════════════════════════════════════════════════════════════════

Modo: Economy.

### Problema
`normalizeFormDados` mapeia camelCase → snake_case só pros campos listados no
`KEY_MAP`. Campo novo num formulário que segue o padrão camelCase, mas não
adicionado ao KEY_MAP, faz a arte sair com `{{campo_snake}}` vazio, sem erro.
Solução: manter o `KEY_MAP` (pros aliases irregulares, ex. isPrivate →
is_private_key) e adicionar uma derivação automática pros casos regulares.

### Arquivo: `artifacts/api-server/src/routes/forms.ts`

LOCALIZAR o final da função `normalizeFormDados` (o trecho do palSvn + o return):
```ts
  for (let i = 1; i <= 4; i++) {
    const k = "palSvn" + i;
    const v = out[k];
    if (typeof v === "string") {
      const lower = v.toLowerCase();
      if (lower === "sim")              out[k] = "Sim";
      else if (lower === "nao" || lower === "não") out[k] = "Não";
    }
  }

  return out;
}
```

SUBSTITUIR por:
```ts
  for (let i = 1; i <= 4; i++) {
    const k = "palSvn" + i;
    const v = out[k];
    if (typeof v === "string") {
      const lower = v.toLowerCase();
      if (lower === "sim")              out[k] = "Sim";
      else if (lower === "nao" || lower === "não") out[k] = "Não";
    }
  }

  // Alias snake_case automático: pra QUALQUER campo camelCase que ainda não
  // tenha um equivalente snake_case, cria o alias derivado (ex.: nomeAssinatura
  // → nome_assinatura). Os casos irregulares continuam cobertos pelo KEY_MAP
  // acima (que já rodou). Apenas ADICIONA chaves — nunca sobrescreve nem remove
  // valores existentes, então é seguro pra quem lê em camelCase.
  for (const key of Object.keys(out)) {
    if (!/[a-z][A-Z]/.test(key)) continue;
    const snake = key.replace(/([a-z])([A-Z])/g, "$1_$2").toLowerCase();
    if (!(snake in out)) out[snake] = out[key];
  }

  return out;
}
```

### Validação 3B
1. `pnpm run build` ok.
2. Campos irregulares continuam certos: `isPrivate=sim` → `is_private_key=private`;
   `fotoPerfilDigital` → `foto_perfil` (via KEY_MAP).
3. Campo regular novo (ex.: enviar `nomeAssinatura`) → passa a existir
   `nome_assinatura` em `dados` automaticamente.
4. Geração de arte que usa `{{nome_assinatura}}` etc. continua preenchendo.

### Não faça (3B)
- Não remover o `KEY_MAP` (ele cobre os aliases irregulares).
- A nova regra só ADICIONA chaves snake quando não existem — não sobrescrever.
- Não alterar o tratamento de `is_private_key` nem do `palSvn`.
