# Hub SVN — Refatoração do Shell (Sidebar Admin + Header Limpo)

## Contexto e objetivo

Hoje as funcionalidades de admin estão fragmentadas entre abas no dashboard e itens soltos no dropdown do usuário. Queremos consolidar tudo numa sidebar visível apenas pra admins, mantendo a experiência de usuário final simples (sem sidebar).

---

## Princípios

- **Usuário final**: header + conteúdo. Sem sidebar (não há itens suficientes pra justificar).
- **Admin**: header + sidebar à esquerda + conteúdo. Sidebar com seções "Minhas solicitações" e "Administração".
- **Páginas internas (formulários, etc.)**: usam o mesmo shell — admin vê sidebar, usuário não.
- **Tela de login**: sem shell (sem header, sem sidebar).

---

## Etapa 1 — Componente do shell

Criar `static/js/shell.js` (ou similar) que renderiza o layout consistente em todas as páginas autenticadas:

```js
window.Shell = {
  /**
   * Renderiza o shell em volta do conteúdo de uma página.
   * Detecta automaticamente se o usuário é admin e renderiza sidebar conforme.
   * 
   * @param {object} options
   * @param {string} options.activeRoute - identificador da rota ativa (pra destacar na sidebar)
   * @param {HTMLElement} options.contentEl - elemento que vai virar o "main" da página
   * @param {function} options.onBeforeNavigate - callback antes de navegar (pra modais de unsaved)
   */
  render({ activeRoute, contentEl, onBeforeNavigate }) {
    // ...
  }
};
```

A função `Shell.render` faz:
1. Cria estrutura: header + (sidebar se admin) + main
2. Injeta o conteúdo passado (`contentEl`) dentro do main
3. Marca o item ativo da sidebar conforme `activeRoute`
4. Configura handlers de navegação que respeitam `onBeforeNavigate` (pra unsaved changes)

## Etapa 4 — Adoção em cada página

Cada página HTML do hub precisa ser refatorada pra usar `Shell.render`. Exemplo de antes/depois:

**Antes** (página `solicitacoes.html`):

```html
<body>
  <header class="hub-header">
    <img src="..." />
    <div class="user-dropdown">...</div>
  </header>
  <div class="page-tabs">
    <button class="tab active">Solicitações gerais</button>
    <button class="tab">Eventos</button>
    <button class="tab admin-only">Painel Admin</button>
    <button class="tab admin-only">Log de atividade</button>
  </div>
  <div class="page-content">
    <!-- conteúdo principal -->
  </div>
</body>
```

**Depois**:

```html
<body>
  <div id="pageContent">
    <!-- Tabs internas só com Solicitações gerais e Eventos -->
    <div class="page-tabs">
      <button class="tab active">Solicitações gerais</button>
      <button class="tab">Eventos</button>
    </div>
    <div class="page-content">
      <!-- conteúdo principal -->
    </div>
  </div>

  <script src="/js/shell.js"></script>
  <script>
    Shell.render({
      activeRoute: 'dashboard',
      contentEl: document.getElementById('pageContent'),
    });
  </script>
</body>
```

**Mudanças importantes:**
- Header e tabs admin saem do HTML da página
- Tabs ADMIN ("Painel Admin", "Log de atividade") são REMOVIDAS — agora vivem na sidebar como rotas próprias
- Tabs USER ("Solicitações gerais", "Eventos") permanecem dentro da página, pois são sub-views do Minhas solicitações

## Etapa 6 — Impersonar dentro de Usuários

Remover o item "Impersonar usuário" do dropdown do header.

Na página `/admin/usuarios`, em cada linha da tabela de usuários, adicionar coluna "Ações" com botão:

```
[👤 Logar como]
```

Click:
1. Confirma com modal: "Deseja logar como Maria Silva? Sua sessão atual será substituída."
2. Se confirmar: chama endpoint existente de impersonar (`POST /admin/impersonate/:userId` ou similar)
3. Redireciona pra `/` (Minhas solicitações) — admin agora vê a UI que aquele usuário vê (sem sidebar, já que está vendo como user comum)
4. Mostrar banner persistente no topo do header: "🎭 Impersonando Maria Silva — [Voltar pro meu usuário]"

O banner é importante: sem ele, o admin pode esquecer que tá impersonando e fazer ações na conta de outra pessoa.

## Etapa 7 — Unsaved changes ao navegar

Páginas com formulários (form-*.html, admin-templates.html, etc.) podem ter mudanças não-salvas. Quando o usuário clica num link da sidebar, deve aparecer o modal de confirmação **antes** de navegar (mesmo padrão do editor de templates).

Implementação:

A página declara que tem unsaved tracking:

```js
Shell.render({
  activeRoute: 'admin-templates',
  contentEl: document.getElementById('pageContent'),
  onBeforeNavigate: (targetUrl) => {
    if (!isDirty) return true;  // pode navegar
    return new Promise((resolve) => {
      showUnsavedModal({
        onConfirm: () => resolve(true),
        onCancel: () => resolve(false),
      });
    });
  }
});
```

`Shell` intercepta clicks em links da sidebar (e do dropdown do user) e chama `onBeforeNavigate` antes de seguir. Se retornar `false` ou Promise resolvida com `false`, cancela a navegação.

## Etapa 8 — Validação

### Smoke tests usuário final

1. Login como usuário sem role admin → vê header simples, SEM sidebar
2. Dashboard com tabs internas Solicitações gerais / Eventos funcionando
3. Click em "Nova solicitação" → vai pra página de seleção de tipos
4. Click num tipo → vai pro formulário, ainda sem sidebar visível
5. Dropdown do user tem apenas "Meu perfil" e "Sair" — nenhum item admin

### Smoke tests admin

1. Login como admin → vê header + sidebar à esquerda
2. Sidebar mostra "Minhas solicitações" + seção "Administração" com 5 itens
3. Click em "Painel Admin" → navega pra página dedicada, item fica destacado na sidebar
4. Click em "Templates de arte" → vai pro editor, sidebar continua visível
5. Botão `«` colapsa a sidebar pra modo ícone-só
6. Em qualquer formulário, sidebar permanece visível
7. Mudanças não-salvas + click na sidebar → modal de confirmação aparece antes de navegar

### Smoke tests impersonar

1. Admin em `/admin/usuarios` → click "Logar como" em uma linha → confirma
2. UI muda: agora admin vê tudo SEM sidebar (como usuário comum)
3. Banner persistente no topo: "🎭 Impersonando [Nome]"
4. Click em "Voltar pro meu usuário" no banner → sessão original restaurada, sidebar volta

### Smoke tests mobile

1. Em viewport < 768px → sidebar fica oculta inicialmente
2. Botão `☰` no header abre a sidebar como drawer
3. Click num item da sidebar → drawer fecha e navega
